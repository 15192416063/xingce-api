# 接入说明(FastAPI + Chroma + DeepSeek)

## 文件
- `xingce_method_kb.json` — 主方法论库,10 个题型完整 entry + 图推/资料分析两个指针 stub。
- `tuxing_tuili_kb.json` — 图形推理拆分库:1 个判型路由(router)+ 12 个维度 entry(数点/数线/数面/数角数素/平移/旋转翻转/遍历/图形运算/对称/开闭曲直/折纸盒/截面三视图)。
- `ziliao_fenxi_kb.json` — 资料分析拆分库:1 个判型路由 + 9 个公式族(formula_entries)+ 8 个速算技巧(speedup_entries),算例已程序验证。
- `system_prompt.md` — 守护层 System Prompt。

## 图推 / 资料分析的二级检索(重点)
这两类不要只取一条,而是'路由 + 维度/技巧'两段注入:

**图推:**
1. 取 `tuxing_tuili_kb.json` 的 `router`(判型口诀+优先级)始终注入。
2. 若题库已标子类型(如'数面''折纸盒'),直接取对应 `entries[].subtype`;没标则把题目+图描述检索 entries 取 top-1~2。
3. system 里拼:router + 命中维度的 method_steps/techniques。

**资料分析:**
1. 按问法(增长率/比重/基期…)命中 `formula_entries` 对应公式族 → 注入其 formulas + method_steps。
2. 看该公式 entry 的 `speedups` 字段(已列出推荐技巧 id),据此从 `speedup_entries` 取对应技巧注入。例如基期题自动带 `zl_s02_cuowei`(错位加减)+`zl_s03_tezheng`(特征数字)。
3. 始终注入 `router.golden_rules` 和 `global_traps`(防百分点/百分比混淆等通病)。

建议把这两个文件的 entry 也存进 Chroma,但保留 `speedups`/`pointer` 等字段做 entry 间的关联跳转。

## 推荐架构:路由 + 检索注入,而非塞大 prompt

### 1. 入库 Chroma(按题型可检索)
把每个 entry 作为一个文档存入 Chroma。
- 建议把 `module + type + keywords + core_idea` 拼成 embedding 文本(检索用),
- `method_steps + techniques + examples + traps + formula_bank` 作为 metadata/payload(注入用)。
- embedding 模型沿用你现有的 BAAI/bge-large-zh-v1.5。

### 2. 题型路由(命中对应方法论)
用户请求某题解析时,有两种命中方式,二选一或叠加:
- **优先用结构化字段**:如果你的题库每道题已标了 module/type,直接按 type 精确取对应 entry(最准,免检索)。
- **没标 type 时用语义检索**:拿题目文本去 Chroma 检索,取 top-1 entry。

### 3. 拼 Prompt
```
system = system_prompt.md 全文
       + "\n\n【本题方法论】\n" + 命中entry的 method_steps/techniques/formula_bank/traps

user   = "题目:" + 题干
       + "\n选项:" + options
       + "\n标准答案(题库):" + answer
       + "\n请按方法论给出解析。"
```
关键:**answer 一定从题库传入 system/user 上下文**,这是"答案以题库为准"原则能生效的前提。

### 4. 多轮对话保持原则
每一轮都把 `system_prompt.md` 作为 system 角色重新带上(DeepSeek 对 system 遵循度好),用户追问放 user。
- 守护原则在 system,不随对话漂移;
- 当前题目的题干/答案/方法论也保留在上下文里,用户追问时 AI 不会脱离本题。

### 5. 常识题兜底(可选但建议)
常识判断是 AI 最易胡编的。两种处理:
- 简单做法:严格限定"只解释已知答案,不确定处注明依据标准答案"(system 已写)。
- 进阶做法:常识题接你的 RAG 知识库,检索到权威知识点再让 AI 解释,降低杜撰风险。

## 后续扩展
- 图推/资料分析已是拆分粒度,新增细分(如数量类再拆'数角'独立检索)按现有 entry 结构续写,字段保持一致。
- 其余模块若要拆细(如逻辑判断拆'翻译推理'与'可能性推理'各自成条),同样照搬结构。
- 申论不在本库范围,如需可另建。
