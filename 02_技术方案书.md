# 行测相似题智能检索系统 — 技术方案书

> 交付对象：开发工程师
> 文档目的：一次性说清产品定位、功能边界、技术方案、提示词、约束条件，开发据此即可动手。
> 架构：公共题库（管理员上传、全员可搜）+ 用户私有题库（仅本人可搜）双库并存。
> 状态：MVP（附可运行 app.py）+ 产品化多租户 SaaS，分两期。

---

## 一、产品定位（决定一切的前提）

**一句话定位：面向考公人群的「行测相似题智能检索」工具，按考点语义找相似题。**

**核心功能（差异化卖点）：**
> 用户给出一道题（或一个主题描述），系统按「考点/规律的语义相似度」返回最相关的若干道题。

例：库里一道题讲「社区垃圾分类」、另一道讲「乡镇河道治理」，字面关键词完全不同，但考点都是「基层治理」——本系统能将二者关联，关键词检索做不到。

### ⭐ 双库架构（重要：题库分两种，必须并存）

系统的题库分两个范围（scope），检索时合并：

| 题库类型 | 谁上传 | 谁能搜 | 作用 |
|---------|-------|-------|------|
| **公共题库**（scope=public） | 管理员（运营方） | 所有用户 | 开箱即用内容，新用户注册即可搜，解决「空壳」问题 |
| **私有题库**（scope=user:用户id） | 用户本人 | 仅本人 | 用户上传自己的资料，仅自己可见 |

**某用户检索时，能搜到的范围 = 公共题库 + 他自己的私有题库。**

技术实现：每条题向量都带 `scope` 元数据标记，检索时按规则过滤：
```
filter = {"scope": {"$in": ["public", "user:当前登录用户id"]}}
```

> ⚠️ **私有题库的隔离是红线**：A 用户永远搜不到 B 用户的私有题。这既是隐私要求，也是合规基础。
> 公共题库则全体可搜，因此**公共题库的内容来源必须干净**（见下方版权说明）。

### ⚠️ 版权与内容来源（合规红线，开发须知悉）

- **私有题库**：用户上传自己的资料、仅自己使用、不共享——平台是「工具提供方」，风险低。
- **公共题库**：由运营方预置、全体用户可搜，**平台对其内容负有直接责任**。因此公共题库**只能放干净来源的题**：① 运营方原创 / AI 生成；② 采购的正版授权题库。
- **严禁**将网上抓取的、培训机构整理的真题（含其答案解析）放入公共题库——这部分受版权保护，机构维权积极，会让整个产品建立在侵权风险上。
- 正式上线前须由专业律师确认版权细节与「用户上传内容的平台责任」边界。

> 注：本方案只负责实现「上传 + 检索」的技术功能；公共题库放什么内容是运营方的内容采购/生产职责，须独立解决合规问题。

## 二、技术原理（RAG 检索增强）

整体是一套针对「找相似题」裁剪过的 RAG 流程，分两个阶段。

### 建库阶段（上传时，一次性）
```
PDF → 提取文字 → 切分成独立题目 → 每题提炼"考点摘要"
   → 用考点摘要计算 embedding 向量
   → 连同原题存入向量库，打 scope 标记：
      管理员上传 → scope=public（公共库）
      用户上传   → scope=user:用户id（私有库）
```

### 检索阶段（每次搜索）
```
用户输入题目/主题 → 同样提炼"考点摘要" → 计算 embedding
   → 在【公共库 + 该用户私有库】范围内做相似度检索
      filter = {"scope": {"$in": ["public", "user:当前用户id"]}}
   → 返回 Top-K 相似题
```

### ⭐ 核心设计决策（务必理解，否则效果会差）

**不要用「整道题原文」算向量，而要用大模型提炼出的「考点摘要」算向量。**

原因：用户要找的是「考点相似」的题，而非「字面相似」的题。整道题的题干情境、选项内容会引入大量噪音，导致两道考点相同的题向量距离反而很远。先用 LLM 把每道题浓缩成「题型+考点+主题」的结构化摘要（例："言语理解-片段阅读-意图判断-经济发展主题"），用这个干净摘要算向量，相似度就能精准落在考点维度。

**检索阶段的用户输入也必须走完全相同的「提炼考点」流程**，保证查询向量和库内向量用同一把尺子衡量，否则不可比。

## 三、功能边界（MVP 明确的取舍）

基于真实试卷样本分析，MVP 阶段做如下取舍。这些不是偷懒，是为了先把核心链路做扎实。

| 题型 | MVP 处理方式 | 说明 |
|------|------------|------|
| 言语理解、常识判断 | ✅ 纳入 | 题干文字完整，考点清晰，语义检索最佳对象 |
| 类比推理 | ✅ 纳入 | 文字短但考点明确，注意切题别误删（见下文坑点） |
| 逻辑判断 | ✅ 纳入 | 题干文字承载考点 |
| 图形推理 | ⏸️ 二期 | 题干几乎无文字，需多模态识图，单独处理（见第七节） |
| 资料分析 | ⏸️ 二期 | 依赖共享材料，单独切出会失真；且 OCR 易混入图表数字噪音 |
| 申论/材料题 | ❌ 不做 | 结构完全不同，不属于「相似题检索」范畴 |

## 四、已验证的切题方案与真实坑点

> 以下结论来自对真实试卷（江苏事业单位卷）的实测，开发可直接复用，能少踩坑。

### 4.1 文字提取
- 文字版 PDF：用 `pdfplumber` 提取，图会被自动跳过（对图是辅助的题无影响）。
- 扫描版 PDF：需先接 OCR（如 PaddleOCR，中文友好）。**真实用户上传的资料格式会非常杂**，扫描版/手机拍照版要有兜底处理或明确提示用户上传文字版。

### 4.2 切题规则（已实测可用）
- 按行首的「数字.」正则切题：`(?m)^(\d{1,3})\.`
- **关键防错**：只保留题号连续递增（1,2,3...）的块，过滤掉材料正文里误判的编号。
- 在「材料处理题」「（一）根据下列材料」等标记处截断，去掉申论和资料分析部分。

### 4.3 ⚠️ 真实坑点（必看）

**坑1：图形题过滤不能只看字数。**
最初用「文字过少」判定图形题，结果把类比推理题（如「司机：驾驶」这种本身就极短的题）误删了。
正确做法：用图形题的**特征话术**判定，含以下任一即判为图形题并过滤：
`"填入问号处"`、`"呈现一定的规律"`、`"从所给的四个选项"`

**坑2：图表数字混入文字流。**
资料分析题的统计图，OCR/提取时会把图里的数字（坐标轴值等）混进正文，污染考点提炼。二期处理资料分析时需清洗。

**坑3：真题常无答案解析。**
样本试卷只有题干和选项，无答案。这不影响「找相似题」（靠题干就能提炼考点），但意味着 MVP 只能返回相似题、不能附答案。产品设计需知悉此局限。

**坑4：PDF 里的图是「多题拼一张」。**
图形题的图常是好几道题拼在一张大图里，无法简单「第N张图对应第N题」。见第七节多模态方案。

## 五、LLM 提示词（直接可用，开发复制即可）

> 模型可替换（OpenAI / 通义 / 智谱 / DeepSeek 均可，建议中文场景用国产模型）。
> 提炼用便宜的小模型即可（如 gpt-4o-mini 同级）；图推识图需多模态模型。

### 5.1 文字题「提炼考点」提示词
```
你是行测命题专家。请用一句话提炼下面这道行测题的核心考点，
格式：题型 + 考点 + 主题。例如"言语理解-片段阅读-意图判断-经济发展主题"。
只输出这一句话，不要解释。

题目：
{题目完整内容}

考点：
```

### 5.2 检索时「用户输入提炼考点」提示词
```
你是行测命题专家。请用一句话提炼下面内容的核心考点，
格式：题型 + 考点 + 主题。只输出这一句话。

内容：
{用户输入的题目或主题}

考点：
```

### 5.3 图形推理「看图说规律」提示词（多模态，二期）
```
你是行测图形推理专家。这张图里有一道或多道图形推理题。
请对每道题，识别它考查的图形规律，并按下面JSON格式输出（只输出JSON，不要解释）：

[
  {"id": "题号", "rule_type": "规律大类", "rule_detail": "具体规律的文字描述"}
]

规律大类只能从这些里选：样式规律 / 位置规律 / 数量规律 / 属性规律 / 空间重构

注意：如果你不确定某道题的具体规律，rule_detail 就写"规律不明确，需人工确认"，
但 rule_type 仍尽量归一个大类。
```

### 5.4 提示词通用约束
- 所有提炼类提示词都要求「只输出结果、不要解释」，便于程序解析。
- 图推提示词强制 JSON 输出，且 `rule_type` 限定枚举值——即使具体规律识别错了，大类对了相似检索仍可用。
- 温度（temperature）：提炼/识别类任务设 0，保证稳定可复现。

## 六、技术约束与配置参数

### 6.1 切块参数
- 行测题按「整题」为单位切，不按固定字数切（题目有天然边界）。
- 若遇超长材料题，TextSplitter 参考值：chunk_size=1500, chunk_overlap=300，分隔符优先 `["\n\n", "\n", "。", "！", "？"]`。

### 6.2 Embedding 模型
- **中文场景必须选中文支持好的模型。** OpenAI `text-embedding-3-small` 够用；国产推荐 `bge` 系列（中文效果优秀，可私有化部署）。
- ⚠️ 建库和检索必须用**同一个** embedding 模型，否则向量空间不一致，检索全乱。

### 6.3 检索参数
- Top-K：默认返回 3-5 道。
- **相似度阈值**：必须设阈值。库中无足够相似的题时，返回「未找到相似题」，而非硬凑低质量结果。具体阈值需用真实数据调（L2 距离越小越相似）。
- **元数据过滤**：检索必须带 `scope` 过滤（公共库 + 本人私有库，见第一/八节），且可叠加「题型」过滤实现「在某题型内找相似」。

### 6.4 模型可替换性
- 代码应将「LLM 调用」「embedding 调用」抽象成可配置模块，方便切换服务商（OpenAI/国产）。不要把某家 API 写死。

## 七、图形推理题方案（二期，多模态路线）

核心思路：**让多模态模型把「图」翻译成「规律描述文字」，翻译后回归标准文字检索流程。**

### 流程
```
含图推题的PDF页 → 渲染成高清图片 → 多模态模型识别"考查什么规律"
  → 输出规律描述(JSON) → 人工抽查修正 → 规律描述算向量入库
  → 之后与文字题统一检索
```

### 关键约束
1. **抠图问题**：PDF 内图推图常「多题拼一张」。推荐方案：用 PyMuPDF 把**整页**渲染成高清图（2倍分辨率，实测 1191x1684 清晰度足够），整页喂给多模态模型，让模型自己按题号区分，避免脆弱的坐标裁剪。
2. **识别准确率有限**：图推规律微妙，多模态模型经常看错。必须有「人工抽查修正」环节。靠 JSON 结构化 + rule_type 枚举兜底：大类对了，相似检索就仍可用。
3. **成本控制**：图推识图调多模态，单题成本是文字题的 30-50 倍。**必须入库时处理一次、存下来复用，绝不能每次检索都实时调用。**

## 八、产品化架构（二期，多租户 SaaS）

> MVP 用单机脚本 + 本地 Chroma 即可验证。以下是要卖会员时的架构。

### 分层
1. **前端/客户端**：上传题目、输入查询、展示相似题结果。管理员端额外有「上传到公共库」入口。
2. **后端 API + 用户系统**：登录鉴权、区分管理员/普通用户、会员/额度校验、请求路由。
3. **入库管线（上传时）**：解析PDF→切题→提炼考点→（图推调多模态）→算embedding→打 scope 标记。贵操作，一次性。
4. **检索管线（搜索时）**：查询提炼考点+算向量→按 scope 过滤检索（公共库+本人私有库）。轻量，极便宜。
5. **向量数据库（多租户）**：换掉单机 Chroma，用 Qdrant/Milvus/Pinecone。每条向量带 `scope` 标签，检索按 scope 过滤。
6. **对象存储**：存上传的原始 PDF/图片。

### ⭐ 隔离规则（核心红线）—— 基于 scope 标记
- 每条向量入库时打 `scope` 标记：管理员上传 → `public`；用户上传 → `user:用户id`。
- 检索时按规则过滤：`{"scope": {"$in": ["public", "user:当前用户id"]}}`
  - 公共库（public）：所有用户可搜。
  - 私有库（user:xxx）：**仅本人可搜，A 用户永远搜不到 B 用户的私有题**。
- 私有隔离满足隐私要求；公共库全体可搜，故其内容来源必须干净（见第一节版权说明）。
- **管理员权限校验**：只有管理员身份能写入 scope=public 的题，必须在后端鉴权，不能仅靠前端控制。

### 入库/检索分离原则
- 所有「贵」的操作（提炼考点、图推多模态）在上传入库时一次性完成，结果落库。
- 用户每次检索只做轻量查询向量化。这是成本可控的关键。

## 九、成本与定价参考（按2026年API价格估算）

> 注：用 OpenAI 价格估算为「成本上限」，国产模型更便宜，真实成本更低。价格易变，上线前需重算。

### 单位成本
| 操作 | 成本 | 频次 |
|------|------|------|
| 文字题入库（提炼+embedding） | ≈ ¥0.0024/题（约四题一分钱） | 一次性 |
| 图推题入库（多模态） | ≈ ¥0.1/题（文字题30-50倍） | 一次性 |
| 用户单次检索 | ≈ ¥0.0024/次（搜400次约一元） | 每次搜索 |

### 典型重度用户测算
传 2000 题（含 200 图推）、月搜 500 次：
- 入库一次性成本：约 ¥26（仅一次）
- 之后每月检索成本：约 ¥1/月

### 定价结论
- **AI 推理成本极低，不是瓶颈。** 月费定 ¥19，毛利率也极高。
- **真正要控的成本**：① 图推多模态入库量（用会员额度限制）；② 向量库/服务器固定月费；③ 对象存储。
- **定价应与「上传额度」挂钩**，而非单纯按时间。建议：免费版限传题数、会员版更高额度、图推题单独限额。

## 十、已交付的 MVP 参考实现

随方案附带一个可运行的最小 Web 应用 `app.py`（Streamlit 单文件），已实现核心功能，技术人员可直接参考或在此基础上扩展：

- **管理员上传**：输入口令 → 上传 PDF → 自动切题、提炼考点 → 存入公共题库（scope=public）。
- **用户检索**：任何人输入题目/主题 → 返回公共库中考点最相似的题，带相似度分数和阈值控制。
- **已内置**：切题逻辑（已在真实试卷验证）、图形题自动过滤、考点提炼提示词、scope 标记机制。
- **预留扩展**：代码注释中已说明如何加入用户私有题库（把 scope 改成 user:用户id，检索 filter 改为 `$in` 公共+私有）。

⚠️ 该 app.py 是**功能验证版**，用本地 Chroma、口令写死。产品化时按本方案改造：
1. 口令登录 → 正规鉴权 + 管理员/用户角色区分（管理员才能写 public）。
2. 本地 Chroma → Qdrant/Milvus 等多租户向量库。
3. 模型可换国产（改 `get_models()`）。

相关文件清单：`app.py`（Web应用）、`step1~4_*.py`（分步参考脚本）、`运行说明_app.md`。

## 十一、开发优先级建议

1. **先做 MVP**：用附带的 `app.py` 把「上传公共库 → 切题 → 提炼考点 → 相似检索」全链路跑通验证。目标是验证「找相似题」的实际效果和用户付费意愿。
2. **验证需求后**再投入：用户私有题库、多租户向量库改造、图推多模态、会员/额度系统、正规登录鉴权。
3. **上线前**：单独确认版权合规细节，并确保公共题库内容来源干净（原创/AI生成/正版采购）。

> 核心提醒：技术实现不是难点，验证「用户是否愿为此付费」才是。建议先用 MVP 拿真实用户试用，再决定是否投入完整 SaaS 工程。
