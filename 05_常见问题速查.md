# 常见问题 / 踩坑速查（FAQ）

> 这些是开发本系统过程中实际会遇到的坑，部分已在真实试卷上验证过。
> 遇到问题先查这里，能省不少来回。

## 切题相关

**Q：切出来的题数明显不对（多了或少了）？**
A：行测卷子排版各家不同。检查切题正则 `^(\d{1,3})\.` 是否匹配你的题号格式——
有的卷子题号是「1、」不是「1.」，有的选项是「A、」不是「A.」。
按你的实际卷子调正则。这是不同来源 PDF 最常见的差异点。

**Q：类比推理题（如"司机：驾驶"）被当成图形题删掉了？**
A：这是我们踩过的坑。不要用「文字太少」判断图形题（类比题本身就短）。
要用特征话术判断：含 `"填入问号处"`、`"呈现一定的规律"`、`"从所给的四个选项"` 才是图形题。

**Q：资料分析题切出来内容很乱、混入一堆数字？**
A：资料分析题依赖前面的共享材料，且统计图里的数字会被提取混入正文。
MVP 阶段建议直接跳过资料分析题，二期再单独处理。

**Q：PDF 提取不出文字 / 提取出来是乱码？**
A：八成是扫描版 PDF（图片拼的，鼠标选不中文字）。pdfplumber 只能提取文字层。
扫描版需要接 OCR（PaddleOCR 中文友好），或提示用户上传文字版。

## 模型 / 向量相关

**Q：检索结果很离谱，找出来的题根本不相似？**
A：按可能性从高到低排查：
1. 建库和检索用了【不同的】embedding 模型 → 向量空间不一致，必然错乱。务必统一。
2. 用了对中文支持差的 embedding 模型 → 换中文模型（如通义 text-embedding-v3 或本地 bge）。
3. 没有用「考点摘要」算向量，而是用了整道题原文 → 噪音太大。回到核心设计：先提炼考点再算向量。

**Q：中文 embedding 该选哪个？**
A：OpenAI 的 text-embedding-3-small 中文够用但非最优。
更好的中文选择：通义 text-embedding-v3（API）、或本地开源 BAAI/bge-large-zh-v1.5（免费、私有化、效果好）。

**Q：相似度阈值（threshold）该设多少？**
A：没有标准值，必须用你的真实数据调。Chroma 默认返回 L2 距离，越小越相似。
方法：入库一批题，拿几道已知相似/不相似的去测，观察分数分布，定一个能区分"相似"和"不相关"的分界值。
一般先从 1.0 试起，再根据效果收紧或放宽。

**Q：换成国产模型后报错？**
A：看「国产模型接入示例.py」。注意：
1. LLM 大多兼容 OpenAI 接口，改 base_url + model 即可。
2. 但 embedding 不一定兼容，各家维度也不同——确认厂商提供 embedding 接口，且全程用同一个。

## 向量库 / 运行相关

**Q：Chroma 报错 / 数据丢失？**
A：MVP 用的本地 Chroma 把数据存在 persist_directory 指定的文件夹。
确认该文件夹有读写权限、没被误删。换电脑/部署时要一起带上这个文件夹。
注意：Chroma 是单机的，多用户并发场景必须换 Qdrant/Milvus（见技术方案书第八节）。

**Q：Streamlit 每次操作都重新加载模型，很慢？**
A：用 @st.cache_resource 缓存模型和向量库实例（app.py 里已经这么做了）。
确认没把模型初始化写在每次点击都执行的地方。

**Q：图推多模态识别经常出错？**
A：这是已知局限，图形规律微妙，模型常看错。应对：
1. 必须有人工抽查修正环节。
2. 靠 rule_type 枚举兜底——大类对了，相似检索仍可用。
3. 多模态贵，入库时处理一次存下来，绝不每次检索都调。

## 成本 / 部署相关

**Q：会不会很费钱？**
A：文字题 AI 成本极低（约每题几厘、单次检索几厘）。真正的成本是：
图推多模态入库（用额度限制）、向量库/服务器月费、对象存储。详见技术方案书第九节。

**Q：API key 怎么管理才安全？**
A：放 .env 文件用环境变量读取，.env 加入 .gitignore（已配好），绝不提交到代码仓库。
