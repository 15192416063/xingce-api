请根据以下详细要求，设计并实现一个基于 Python 的「行测相似题智能检索系统」。该系统的核心能力是：用户给出一道行测题（或一个主题描述），系统按「考点的语义相似度」从题库中找出最相关的若干道题——这是普通关键词搜索做不到的差异化卖点。请给出完整的设计方案、核心数据结构、关键代码框架及实现逻辑，确保系统具备良好的可扩展性和可维护性。

---

### 〇、核心概念先行（理解这一条，整个系统才不会做歪）

本系统是一套针对「找相似题」裁剪过的 RAG（检索增强）应用。最关键的设计决策：

**不要用「整道题原文」去计算向量，而要先用 LLM 把每道题提炼成「考点摘要」，再用考点摘要计算向量。**

原因：用户要找的是「考点相似」的题，不是「字面相似」的题。题干情境和选项内容会引入大量噪音，导致两道考点相同但表述不同的题向量距离很远。先把每道题浓缩成「题型+考点+主题」的结构化摘要（例："言语理解-片段阅读-意图判断-经济发展主题"），用这个干净摘要算向量，相似度就能精准落在考点维度。检索时，用户输入也必须走完全相同的「提炼考点」流程，保证查询与库内用同一把尺子衡量。

### 一、总体架构要求

1. **双库架构（最重要的架构特征，必须并存）**
   系统题库分两个范围（scope），检索时合并：
   - **公共题库**（scope=public）：由管理员上传，所有用户都能搜。作用是「开箱即用」内容，新用户注册即可使用，避免空壳。
   - **用户私有题库**（scope=user:用户id）：用户上传自己的资料，仅本人可搜。
   - 某用户检索时，可搜范围 = 公共题库 + 他自己的私有题库。
   - 实现方式：每条题向量带 `scope` 元数据标记，检索时按 `{"scope": {"$in": ["public", "user:当前用户id"]}}` 过滤。
   - **隔离红线**：A 用户永远搜不到 B 用户的私有题，必须在后端强制过滤，不能仅靠前端控制。
   - **管理员权限**：只有管理员身份能写入 scope=public 的题，必须后端鉴权。

2. **多模型支持（避免与具体模型耦合）**
   - 系统需调用两类模型：① LLM（用于提炼考点，用便宜的小模型即可）；② Embedding 模型（用于算向量，中文场景务必选中文支持好的模型）。
   - 用户/管理员可配置使用哪家服务（OpenAI、通义千问、智谱、DeepSeek、本地模型等），通过统一接口调用，方便切换。
   - 采用适配器模式：定义抽象基类，为不同服务实现具体适配器。业务逻辑不得写死某一家 API。
   - ⚠️ 约束：建库和检索必须使用**同一个** embedding 模型，否则向量空间不一致，检索结果全乱。

3. **PDF 上传与自动切题**
   - 支持上传文字版 PDF，自动提取文字、切分成一道道独立的题。
   - 切题规则（已在真实试卷验证，可直接复用）：
     - 按行首的「数字.」正则切题：`(?m)^(\d{1,3})\.`
     - 只保留题号连续递增（1,2,3...）的块，过滤材料正文里误判的编号。
     - 在「材料处理题」「（一）根据下列材料」等标记处截断，去掉申论和资料分析部分。
   - 扫描版 PDF 需接 OCR（如 PaddleOCR）作为兜底，或明确提示用户上传文字版。

4. **题型范围与取舍（MVP 阶段明确）**
   - ✅ 纳入：言语理解、常识判断、类比推理、逻辑判断——题干文字承载考点，语义检索的最佳对象。
   - ⏸️ 二期：图形推理（题干几乎无文字，需多模态识图，单独处理，见第三节）、资料分析（依赖共享材料，单独切出会失真）。
   - ❌ 不做：申论/材料题（结构不同，不属于相似题检索范畴）。

### 二、数据结构设计要求

请为以下核心实体设计合适的类或数据表结构（向量数据用向量库，元数据可用关系库/JSON）：

- `Question`（题目）：id、原始内容(content)、考点摘要(topic)、题型(type)、scope(public 或 user:xxx)、来源文件、入库时间。
- `QuestionVector`（题目向量）：对应 question_id、向量值、scope 标记（用于检索过滤）。
- `User`（用户）：id、角色(管理员/普通用户)、会员等级、上传额度。
- `ModelConfig`（模型配置）：服务商、API 端点、密钥、模型名、参数（温度、维度等）。区分 LLM 配置和 Embedding 配置。
- `UploadLog`（上传记录）：用户、文件、切出题数、入库题数、过滤掉的题数、时间，用于审计和额度统计。

> 注意：Question 的 `topic`（考点摘要）字段是系统的核心——它是参与向量计算的内容，content 仅作展示和存储。

### 三、核心功能实现要点

1. **切题模块（Parser）**
   - 输入 PDF 路径，输出一道道题的纯文本列表。
   - 包含上述切题正则、题号连续性校验、申论/资料分析截断逻辑。
   - **图形题过滤**：含以下任一特征话术的题判定为图形题并跳过：`"填入问号处"`、`"呈现一定的规律"`、`"从所给的四个选项"`。
   - ⚠️ 已知坑：不能用「文字过少」过滤图形题，否则会误删类比推理题（如「司机：驾驶」本身就很短）。必须用上述特征话术过滤。

2. **考点提炼模块（TopicExtractor）**
   - 输入一道题的完整文本，调用 LLM 输出一句话考点摘要。
   - 建库和检索复用同一套提炼逻辑。
   - 提示词（直接可用）：
     ```
     你是行测命题专家。请用一句话提炼下面这道行测题的核心考点，
     格式：题型 + 考点 + 主题。例如"言语理解-片段阅读-意图判断-经济发展主题"。
     只输出这一句话，不要解释。

     题目：
     {题目完整内容}

     考点：
     ```
   - 约束：temperature 设 0，保证稳定；要求「只输出结果、不要解释」便于程序解析。

3. **建库模块（Indexer）**
   - 流程：PDF → 切题 → 逐题提炼考点 → 用考点摘要算 embedding → 连同原题、题型、scope 标记存入向量库。
   - 管理员上传：scope=public；用户上传：scope=user:用户id。
   - 入库前做上传额度校验（按用户会员等级）。

4. **检索模块（Searcher）**
   - 输入：用户的题目/主题文本、当前用户id、Top-K、相似度阈值。
   - 流程：输入文本走相同的考点提炼 → 算 embedding → 在「公共库 + 本人私有库」范围内做相似度检索。
   - **必须设相似度阈值**：库中无足够相似的题时，返回「未找到相似题」，而非硬凑低质量结果（L2 距离越小越相似，阈值需用真实数据调）。
   - **可叠加题型过滤**：实现「在某题型内找相似」（元数据过滤 + 向量检索）。

5. **入库/检索分离原则（成本关键）**
   - 所有「贵」的操作（提炼考点、图推多模态）在上传入库时一次性完成，结果落库。
   - 用户每次检索只做轻量的查询向量化。绝不能每次检索都重新调用昂贵操作。

### 四、图形推理题方案（二期，多模态路线）

核心思路：让多模态模型把「图」翻译成「规律描述文字」，翻译后回归标准文字检索流程。

- **抠图**：PDF 内图推图常「多题拼一张」，无法简单一题一图。推荐用 PyMuPDF 把含图推题的**整页**渲染成高清图（2 倍分辨率），整页喂给多模态模型，让它按题号区分，避免脆弱的坐标裁剪。
- **识别提示词**（强制 JSON 输出）：
  ```
  你是行测图形推理专家。这张图里有一道或多道图形推理题。
  请对每道题，识别它考查的图形规律，并按下面JSON格式输出（只输出JSON）：
  [{"id": "题号", "rule_type": "规律大类", "rule_detail": "具体规律描述"}]
  规律大类只能从这些里选：样式规律 / 位置规律 / 数量规律 / 属性规律 / 空间重构
  若不确定具体规律，rule_detail 写"规律不明确，需人工确认"，但 rule_type 仍尽量归类。
  ```
- **约束**：① 识别准确率有限，必须有人工抽查修正环节；靠 rule_type 枚举兜底——大类对了相似检索仍可用。② 多模态调用是文字题的 30-50 倍成本，必须入库时处理一次、存下来复用。

### 五、界面与交互要求

提供两类界面：

1. **用户检索界面**（所有人）
   - 输入框：粘贴一道题或描述一个主题。
   - 参数：返回数量（Top-K）、相似度阈值。
   - 结果区：展示识别到的考点、每道相似题的相似度分数和原文。
   - 无结果时明确提示「未找到足够相似的题」。

2. **管理员上传界面**（需鉴权）
   - 身份验证（MVP 可用口令，产品化换正规登录 + 角色区分）。
   - 上传 PDF → 显示切题进度 → 提炼考点入库 → 反馈入库结果（切出多少、过滤多少、入库多少）。
   - 题目进入公共题库（scope=public）。

3. **用户私有上传界面**（二期）
   - 普通用户上传自己的 PDF，scope=user:本人id，仅自己可搜。

- 所有自动操作（切题、过滤、入库）都应有进度提示或日志。

### 六、实现语言与依赖

- Python 3.10+。
- PDF 处理：`pdfplumber`（文字提取）、`PyMuPDF`（图推渲染，二期）。
- LLM/Embedding：`langchain` 系列 + 对应服务商 SDK（OpenAI / 通义 / 智谱 / DeepSeek）。
- 向量库：MVP 用 `chromadb`（本地，单文件即可验证）；产品化换 Qdrant/Milvus/Pinecone（支持多租户、高并发）。
- 界面：MVP 推荐 `streamlit`（单文件快速出原型）；产品化用正式前后端分离。
- 数据存储：元数据用 `sqlite3` 或 `tinydb`（MVP）。
- 代码遵循模块化：界面、业务逻辑（切题/提炼/建库/检索）、数据访问、模型适配器分离。
- 提供 README 说明运行和扩展方式。

### 七、版权与合规约束（红线，开发须知悉）

- **私有题库**：用户上传自己的资料、仅自己用、不共享——平台是工具提供方，风险低。
- **公共题库**：全员可搜、平台担直接责任，**内容来源必须干净**：① 运营方原创 / AI 生成；② 采购的正版授权题库。
- **严禁**将网上抓取的、培训机构整理的真题（含其答案解析）放入公共题库——受版权保护，风险极高。
- 本系统只负责实现「上传 + 检索」的技术功能；公共题库放什么内容是运营方的内容生产职责，须独立解决合规问题。
- 正式上线前须由专业律师确认版权细节与平台责任边界。

### 八、成本与性能约束

- 文字题入库成本极低（约每题几厘钱），用户单次检索成本极低（搜数百次约一元）——AI 成本不是瓶颈。
- 真正要控的成本：① 图推多模态入库量（用会员额度限制）；② 向量库/服务器固定开销；③ 对象存储。
- 因此：上传额度按会员等级限制，图推题单独限额；入库的贵操作只做一次并落库复用。

### 九、开发优先级建议

1. **先做 MVP**：用 Streamlit 单文件实现「管理员上传公共库 → 切题 → 提炼考点 → 相似检索」全链路，本地 Chroma 即可。目标是验证「找相似题」的实际效果和用户付费意愿。
2. **验证需求后**再投入：用户私有题库、多租户向量库改造、图推多模态、会员/额度系统、正规登录鉴权。
3. **上线前**：确认版权合规，确保公共题库内容来源干净。
